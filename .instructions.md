# Project Instructions for Go Application

## API Communication
- **Endpoints**:
  - `/api/motors`: Set motor speeds. Accepts query parameters `a` and `b` (integer range: -255 to 255). Values below ±200 may not move the motors effectively.
  - `/api/servo`: Set servo angle. Accepts query parameters `id` (1 or 2) and `angle` (0 to 180).
  - `/api/stop`: Stop motors immediately. No parameters required.
  - `/api/status`: Retrieve current state. Returns JSON with `motorA`, `motorB`, `servo1`, and `servo2` states.
  - `/api/control`: Set motors and servos in one call. Accepts optional query parameters `a`, `b`, `s1`, and `s2`. Returns full state and battery information (`pct`, `v`, `mv`).
  - `/api/battery`: Retrieve battery status. Returns `pct` (0-100), `v` (volts), and `mv` (millivolts).
  - `/api/wifi`: Change Wi-Fi credentials. Accepts `ssid` (1-32 chars) and `pass` (8-63 chars). AP restarts 500ms after response.

- **HTTP Client**: Use Go's `net/http` package for API communication.
- **Retries**: Implement retry logic for transient errors.
- **Timeouts**: Set appropriate timeouts for HTTP requests.

### Mapping Controller Inputs to API Commands
- **Joystick Axes**:
  - Map left joystick vertical axis to `a` (motor A speed).
  - Map right joystick vertical axis to `b` (motor B speed).
- **Buttons**:
  - Map specific buttons to servo control (e.g., `id=1` for left trigger, `id=2` for right trigger).
  - Use button presses to trigger `/api/stop`.
- **Combos**:
  - Allow combinations of joystick and button inputs to call `/api/control` with multiple parameters.
- **Dead Zones**:
  - Implement dead zones for joystick inputs to avoid unintentional movements.

## Input Modes

### Xbox Controller Mode (inputMode: xbox)
- **Windows**: Uses native XInput API for direct Xbox controller support. No external dependencies required.
- **Linux**: XInput is not supported. Use joystick devices via `/dev/input/js*` (requires implementation).
- **Tank Mode Control**: 
  - Left stick Y-axis controls motor A (left motor)
  - Right stick Y-axis controls motor B (right motor)
  - Push both sticks forward → Move forward
  - Push both sticks backward → Move backward
  - Push left forward + right backward → Spin right
  - Push right forward + left backward → Spin left
- **Servo Control**:
  - L1 (Left Shoulder/LB) → Servo 1 to min position
  - L2 (Left Trigger/LT) → Servo 1 to max position
  - R1 (Right Shoulder/RB) → Servo 2 to min position
  - R2 (Right Trigger/RT) → Servo 2 to max position
- **Polling**: Reads controller state every 50ms.
- **Deadzone**: Automatically applies standard Xbox deadzone values (7849 for left stick, 8689 for right stick).
- **Error Handling**: Handles disconnections gracefully with automatic reconnection attempts.

### Keyboard Mode (inputMode: keyboard)
- **Tank Mode Control** (customizable via config.yaml):
  - Default: **W** → Motor A forward (220 speed)
  - Default: **S** → Motor A backward (-220 speed)
  - Default: **I** → Motor B forward (220 speed)
  - Default: **K** → Motor B backward (-220 speed)
  - Default: **Space** → Stop all motors immediately
- **Servo Control** (customizable via config.yaml):
  - Default: **A** → Servo 1 to min position
  - Default: **D** → Servo 1 to max position
  - Default: **J** → Servo 2 to min position
  - Default: **L** → Servo 2 to max position
- **Customization**: All keys can be changed in `config.yaml` under the `keyboard` section. The application will display your configured keys on startup.
- **Supported Keys**: Any letter (a-z), space, enter, tab, esc, arrow keys (up, down, left, right).
- **Behavior**: 
  - Motors stop immediately when keys are released (no momentum)
  - Servos return to center position (90°) when keys are released
  - Hold keys to maintain movement or servo position
- **Polling**: Reads keyboard state every 100ms.
- **Cross-platform**: Works on Windows, Linux, and macOS.

## Configuration File (config.yaml)
The application uses a YAML configuration file to customize motor and servo behavior:

```yaml
# Input mode: "xbox" for Xbox controller, "keyboard" for keyboard controls
inputMode: xbox

# Motor configuration
motors:
  # Invert motor A if wired backwards (true/false)
  invertA: false
  
  # Invert motor B if wired backwards (true/false)
  invertB: false

# Servo configuration
servos:
  servo1:
    min: 0    # Minimum angle (degrees)
    max: 180  # Maximum angle (degrees)
  servo2:
    min: 0
    max: 180

# Keyboard controls (only used when inputMode is "keyboard")
keyboard:
  motorA:
    forward: w
    backward: s
  motorB:
    forward: i
    backward: k
  servo1:
    min: a
    max: d
  servo2:
    min: j
    max: l
  stop: space  # Can use: space, enter, tab, esc, up, down, left, right, or any letter
```

**Input Mode**: Choose between `xbox` (Xbox controller) or `keyboard` (keyboard controls).

**Motor Inversion**: If a motor runs in the wrong direction due to incorrect wiring, set `invertA` or `invertB` to `true` instead of re-soldering.

**Servo Range**: Customize servo angle limits if your servos have physical constraints or you want to limit movement range.

**Keyboard Key Mappings**: Fully customizable keyboard controls. You can assign any letter key (a-z) or special keys (space, enter, tab, esc, arrow keys) to any function.

## Running the Application
- **Windows** (recommended): 
  ```powershell
  # Run with default config.yaml (Xbox mode)
  .\robocombat.exe -host 192.168.4.1 -debug
  
  # Run with keyboard mode (edit config.yaml first: inputMode: keyboard)
  .\robocombat.exe -host 192.168.4.1
  
  # Run with custom config file
  .\robocombat.exe -host 192.168.4.1 -config myconfig.yaml
  ```
- To build for Windows from WSL/Linux:
  ```bash
  GOOS=windows GOARCH=amd64 go build -o robocombat.exe
  ```
- To build for Linux: `go build -o robocombat`

**First Time Setup**:
1. Copy the executable and `config.yaml` to the same directory
2. Edit `config.yaml` to match your hardware setup:
   - Set `inputMode` to `xbox` or `keyboard`
   - Configure motor inversions if needed
   - Adjust servo ranges if necessary
3. Run the application

## Additional Notes
- **Dual Input Support**: The application supports both Xbox controller (XInput) and keyboard input, selectable via `config.yaml`.
- **Fully Customizable Keyboard**: In keyboard mode, all keys can be remapped in `config.yaml` to suit your preferences or keyboard layout.
- **XInput Support**: The Xbox controller mode uses native Windows XInput API. This provides the best compatibility and performance on Windows.
- **Keyboard Mode**: Keyboard mode is cross-platform and works on Windows, Linux, and macOS. It's ideal when an Xbox controller is not available.
- **Controller ID**: In Xbox mode, use `-controller` flag to specify which controller to use (0-3 for up to 4 controllers).
- **Debug Mode**: Use `-debug` flag to see detailed input (controller buttons/axes or keyboard keys) and commands sent to the robot.
- **No External Dependencies for Xbox**: The Windows version requires no additional libraries or drivers for Xbox controller - XInput is built into Windows.
- **Configuration File**: If `config.yaml` is not found, the application will use default settings (Xbox mode, default keyboard keys, no motor inversion, servos 0-180°). Create one to customize behavior without recompiling.
- **Motor Wiring Fix**: If motors run backwards, simply edit `config.yaml` and set `invertA` or `invertB` to `true` - no need to re-solder!
- **Input Mode Selection**: Change `inputMode` in `config.yaml` to switch between Xbox controller and keyboard without recompiling.
- **Key Display on Startup**: When using keyboard mode, the application displays your configured key mappings at startup.
